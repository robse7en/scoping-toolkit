# Deterministic Manifest Synchronization Design

## Goal

Replace the model-generated `/sync-manifest` transformation with an executable,
cross-platform script whose validation and output are reproducible. Keep task-file
frontmatter as the source of truth and preserve the existing manifest format.

## Scope

This change adds a dependency-free Python manifest generator, automated tests, and
the command/documentation changes required to install and invoke it. It does not
change the task lifecycle, task frontmatter schema, or manifest presentation.

## Architecture

The generator will live at
`docs/project-scope/_scripts/sync_manifest.py`, alongside the project-scoped
templates it consumes. Keeping it project-scoped makes invocation consistent
whether Claude commands are installed personally or in the target repository.

The script will use only the Python standard library. It will accept the target
repository root and an optional timestamp override. The timestamp override is a
test seam; normal command execution uses the current UTC time.

Processing has four isolated stages:

1. Discover task files under `docs/project-scope/phases/*/tasks/*.md`.
2. Parse and validate the supported YAML-frontmatter subset.
3. Build a normalized in-memory manifest model and render stable Markdown.
4. Write to a temporary file and atomically replace `manifest.md` after every
   validation and render step succeeds.

## Validation

Generation fails without modifying the existing manifest when any of these are
true:

- `architecture.md` is missing or has no positive integer version;
- no task files are found;
- a task lacks a required frontmatter field;
- a task ID is duplicated;
- `phase` or `architectureVersion` is not a positive integer;
- a task's `phase` does not match its `phase-N-<slug>` parent folder;
- `status` is not one of `pending`, `in-progress`, `review`, `done`, or `blocked`;
- `dependsOn` or `touchesFiles` is not a list;
- a dependency references an unknown task ID;
- a task depends on itself; or
- the dependency graph contains a cycle.

Blocked-task reasons will be extracted mechanically from the first meaningful
line in the task's `Architecture Conflict` section. If no reason is present, the
manifest will state `Not documented` rather than asking a model to infer one.

## Determinism

Tasks will be ordered by numeric phase and then task ID. Human-readable phase
names will be derived from each `phase-N-<slug>` folder by replacing hyphens with
spaces and title-casing the result. Phase summaries and blocked-task rows will use
the same normalized ordering. Given identical input files and an identical
timestamp argument, the generated bytes will be identical.

The script will print a single summary line containing total, done, and blocked
counts. It will return a non-zero exit code with a specific error message on
validation or write failure.

## Command Integration

`.claude/commands/sync-manifest.md` will become a thin orchestration layer:

1. Locate the project-scoped script.
2. Run it with the repository root.
3. Relay its summary or exact failure.

The command will not parse task files, choose blocked reasons, or write the
manifest itself.

Installation documentation will copy both `_templates` and `_scripts` into each
target project. Python 3.9 or newer will be documented as the script prerequisite.

## Testing

Tests will use Python's `unittest` and temporary repository fixtures. Coverage
will include:

- stable rendering and ordering from valid task files;
- phase/status counts and blocked-reason extraction;
- byte-identical output with a fixed timestamp;
- validation of malformed fields, duplicate IDs, unknown dependencies, and
  self-dependencies;
- preservation of an existing manifest after validation failure;
- command documentation invoking the script rather than performing model-driven
  rendering.

The existing PowerShell agent-contract validation remains part of full
verification.

## Error Handling

Expected input errors will be reported without a traceback and return exit code
1. Unexpected programming errors may retain a traceback during development, but
the command wrapper must still report failure and must not claim the manifest was
updated.

## Non-Goals

- Supporting arbitrary YAML beyond the task schema's current scalar and inline
  list forms.
- Adding a package manager, third-party YAML parser, or compiled executable.
- Automatically changing task state or architecture versions.
- Redesigning the manifest layout or task schema.
