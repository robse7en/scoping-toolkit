# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

A workflow toolkit — not an application to build or run. It consists entirely of Markdown and YAML files: slash command definitions, subagent definitions, and document templates. There is no build step, no test suite, and no runtime. Work here means editing `.md`/`.yml` files and understanding how the pieces fit together.

## Repository structure

```
.claude/
  agents/      # 9 subagent definitions (invoked by commands, not directly by users)
  commands/    # 6 slash command definitions (user-facing entry points)
docs/project-scope/
  _templates/  # Schema files for documents the workflow generates in target projects
.serena/
  project.yml  # Serena MCP tool configuration for this repo
```

The toolkit is **installed into other projects**, not run inside this repository itself. `.claude/agents/` and `.claude/commands/` can be installed either personally (`~/.claude/`) or per-project (`.claude/` in the target repo). The `docs/project-scope/_templates/` directory is always installed per-project into the repo being scoped.

## The workflow model

The toolkit separates two phases that must not bleed together:

1. **Scoping** (once, up front): `/scope new|extend` drives a sequential pipeline of subagents — research → interview → feature suggestion → architecture → planning → task-writing → review — with explicit human checkpoints between each stage.

2. **Implementation** (many short sessions): `/implement <task-id>` in a fresh session loads only the minimal context needed (constraints, architecture, decisions, one task file). Each task is designed to be self-contained enough that a new session with no prior memory can execute it.

Progress is tracked entirely in YAML frontmatter in task files. `manifest.md` is a deterministic rollup generated by `/sync-manifest` — it is never hand-edited.

## Slash commands

| Command | What it does |
|---|---|
| `/scope new "<desc>"` | Full pipeline: research → interview → features → architecture → plan → tasks |
| `/scope extend "<desc>"` | Same pipeline but starts with `codebase-analyst` instead of `research-agent` |
| `/implement <task-id>` | Implement one task in a fresh session; stops at unmet dependencies or architecture conflicts |
| `/qa <task-id>` | The only path to `status: done`; invokes `qa-reviewer` |
| `/amend-architecture <task-id>` | Resolves a `blocked` task by invoking `architect-agent` to amend `architecture.md` |
| `/sync-manifest` | Regenerates `manifest.md` from task frontmatter; run after any status change |
| `/status` | Reads `manifest.md` and prints a human-friendly progress summary |

## Single-writer rule (critical)

Each file has exactly one agent permitted to write it. No exceptions:

| File | Only writer |
|---|---|
| `architecture.md` | `architect-agent` |
| `decisions.md` | `architect-agent` |
| `manifest.md` | `/sync-manifest` command |
| `status: done` in any task | `qa-reviewer` |

`/implement` sessions may write to task files (status, Session Log) but must never set `status: done`. If a coding session encounters an architecture mismatch, it sets `status: blocked`, documents the conflict in the task's "Architecture Conflict" section, and stops — it does not improvise a workaround.

## Task status lifecycle

```
pending → in-progress → review → done
                │                  │
          (architecture        (/qa reject)
           mismatch)               │
                └──→ blocked ←─────┘
                      ↑
              (/amend-architecture resolves → pending)
```

## Task file frontmatter fields

Every task under `docs/project-scope/phases/*/tasks/*.md` uses this frontmatter:

```yaml
id: P1-001
phase: 1
title: ""
status: pending          # pending | in-progress | review | done | blocked
dependsOn: []            # task ids that must be done before this task can start
touchesFiles: []         # used by scope-reviewer for parallel-safety checks
parallelSafe: false      # set by scope-reviewer only, never by task-writer
architectureVersion: 1   # version of architecture.md this task was written against
```

`parallelSafe` is set by `scope-reviewer` after checking `touchesFiles` overlap — task-writer never sets it.

## Generated files (live in the target project, not this repo)

| File | Generated by | Rule |
|---|---|---|
| `research-summary.md` | `research-agent` | New projects only |
| `existing-system-context.md` | `codebase-analyst` | Extension projects only |
| `constraints.md` | `interviewer`, `feature-suggester` | Binding; no open questions before architect runs |
| `architecture.md` | `architect-agent` | Versioned; `task-writer` and `/implement` conform to it |
| `decisions.md` | `architect-agent` | Append-only ADR log |
| `manifest.md` | `/sync-manifest` | Never hand-edit |
| `phases/phase-N-*/tasks/*.md` | `task-writer`; edited by `/implement`, `/qa`, `scope-reviewer` | One file per task |

## Editing the toolkit itself

When modifying agent definitions (`.claude/agents/*.md`) or command definitions (`.claude/commands/*.md`):

- Command files use YAML frontmatter (`description`, `argument-hint`, `disable-model-invocation`, `allowed-tools`) followed by the instruction body in Markdown.
- Agent files follow the same pattern. Subagents are invoked by name (e.g. `research-agent`) from commands — the name matches the filename without `.md`.
- Changes to agents/commands take effect only in new Claude Code sessions; sessions already running won't pick up mid-session changes.
- Template files in `docs/project-scope/_templates/` define the exact schema downstream agents must conform to. Changing a template's structure requires reviewing all agent definitions that write or read that file type.
